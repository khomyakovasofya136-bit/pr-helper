# Режим: АДАПТАЦИЯ (главный режим адаптера)

Берём готовую статью и переписываем её под конкретное издание с учётом стиля, длины, угла, рубрики, ЦА.

## Когда вызывается

- Соня даёт готовую статью (.md из `inputs/` или вставкой) + название издания
- Опционально: ТЗ от редакции (если редакция прислала свои требования)
- Опционально: свои тезисы, углы, целевая аудитория

## Что подгружает

Перед началом работы:
1. `pr-helper/shared/zerocoder-context.md` — бренд, спикеры, риск-зоны
2. `promts/style-reference.md` — образцы стиля
3. `promts/editorial-criteria.md` — критерии и антипаттерны
4. `promts/brand-block-pattern.md` — обязательная структура
5. **Google Sheets, лист `Адаптация`** (через `pr-helper/mediabase/sheets.py`) — параметры издания: `стиль_формат, лимит_знаков, рубрики, угол_подачи_основной, ца_дефолт, особые_требования, спикер_основной, последний_ресёрч, комментарий_для_адаптера`. Также получаем `slug` для построения локальных путей.
6. `publications/<slug>/rules.md` — длинные правила и требования
7. `publications/<slug>/research/<последний>.md` — последний ресёрч
8. 1-2 файла из `publications/<slug>/examples/` — эталоны Зерокодера

### Как читать Sheets

```bash
python -c "
import sys; sys.path.insert(0, 'C:/Users/sofit/клодик/pr-helper/mediabase')
import sheets, json
print(json.dumps(sheets.get_adaptation_params('vn.ru'), ensure_ascii=False, indent=2))
"
```

Если в Sheets нет издания — спросить Соню:
> Издание `<site>` не найдено в листе `Адаптация`.
> Добавить запись? Какие параметры (стиль, лимит, рубрики, угол, ЦА, спикер)?

## Алгоритм работы

### Шаг 1. Собрать контекст

- Прочитать всё из списка «Что подгружает»
- Проверить дату последнего ресёрча издания:
  - **< 30 дней** — использовать молча
  - **30-90 дней** — спросить: «Использовать сохранённый ресёрч или сделать свежий?»
  - **> 90 дней или нет ресёрча** — сделать свежий (см. шаг 2)

### Шаг 2. Свежий ресёрч издания (если нужен)

Через WebFetch/WebSearch собрать по сайту издания:
- Топ-5 свежих статей по ИИ
- Все статьи Зерокодера в этом издании (если есть)
- Структура сайта и рубрикатор
- Формат подачи (новости/колонки/интервью/лонгриды)
- Уровень подготовленности аудитории
- Какой формат статей по нашей теме: соотношение практики и теории
- Уровень экспертизы (поверхностно / глубокий разбор / с цифрами)
- Возможности для демонстрации бренда Зерокодера
- Заголовки топ-5 статей по теме
- Структура топ-5 статей по теме

Результат сохранить в `publications/<slug>/research/YYYY-MM-DD.md`.
После — обновить `последний_ресёрч` в Sheets (с подтверждения Сони).

### Шаг 3. Уточнить у Сони — два вопроса

**Вопрос 1 — параметры из базы:**

> Из базы по [издание]:
> - **ЦА**: [значение из rules.md/Sheets]
> - **Формат**: [значение]
> - **Длина**: [значение]
> - **Угол подачи**: [значение]
> - **Особые требования**: [значение, если есть — например, docx-шаблон для Т-Бизнес Секретов]
> - **Спикер**: [значение]
>
> Это всё ещё актуально для текущей статьи? Какие правки?

**Вопрос 2 — ЦА:**

> По ЦА: типовая ЦА для этого издания — **[X]**.
> Подходит, или для этой статьи берём другую? Если другую — напиши какую.
>
> Типовой список ЦА: см. `publications/audience-types.md`.

### Шаг 4. Адаптация текста

С учётом подтверждённых параметров и контекста (включая ТЗ редакции, если есть) — переписать статью:

- Заголовок и подзаголовки — под формат издания
- Лид — по одному из 3 типов конфликта (см. style-reference)
- Брендовый блок + анонс — обязательно (см. brand-block-pattern)
- Длина — в пределах лимита издания
- Стиль и плотность — по style-reference
- 10 критериев — все соблюдены (см. editorial-criteria)
- Антипаттерны — флагировать, исправлять

### Шаг 5. Выход — формат ответа

```
1. Адаптированная статья
   [полный текст]

2. Список изменений с обоснованием
   - Заголовок изменён: [было] → [стало]. Причина: [...]
   - Лид: использован тип конфликта №[X]. Причина: [...]
   - Сокращено на [N] знаков для попадания в лимит [издание]
   - Усилен мост к региональному читателю в блоке [...]
   - Добавлен крючок «[X]» в [месте]
   - Бренд Зерокодера переписан под формулу «проблема → опыт → вывод»
   - [и т.д. для всех значимых правок]

3. Что проверить вручную
   - Цифры и источники: [список меток нужен источник]
   - Спорные формулировки: [перечислить]
   - Англицизмы, требующие пояснения: [перечислить]
   - Цитаты, которые надо подтвердить у спикера: [перечислить]
```

### Шаг 6. Опциональный шаг — обновление Sheets

После выдачи статьи:

> Обновить лист `Адаптация` по [издание]?
> Что предлагается записать:
> - последний_ресёрч: 2026-06-13
> - комментарий_для_адаптера: [новый пункт, если выявили нюанс при адаптации]
> - [и т.д. для всех полей, которые могли измениться]
>
> Подтвердить (да/нет/правки)?

Если Соня подтверждает — записать через `sheets.update_adaptation(site, **updates)`.

Касания (отправлено в редакцию, дата касания, новая тема) — это **лист `Касания`**, не `Адаптация`. Обновляет pitch-агент или CRM. Адаптер туда не пишет.

## Куда сохраняется результат

Файл по нейминг-конвенции: `outputs/[ГГГГ-ММ]_[slug]_[тема]_adapted.md`

Пример: `outputs/2026-06_vn_metrikokratiya_adapted.md`

---

## Что НЕ делать

- Не писать статью с нуля — адаптер только адаптирует готовое
- Не убирать брендовый блок + анонс (см. brand-block-pattern)
- Не предполагать ЦА — всегда спрашивать (см. Шаг 3, вопрос 2)
- Не использовать ресёрч старше 90 дней без свежения
- Не обновлять Sheets без подтверждения Сони
- Не писать в лист `Касания` — это работа pitch и CRM, не адаптера
