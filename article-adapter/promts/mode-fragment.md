# Режим: ПРАВКА ФРАГМЕНТА (точечная микро-работа)

Когда нужно поправить один абзац / блок / заголовок — не весь текст. Главное: вернуть исправленный фрагмент с маркерами проблем и пояснением каждой правки.

## Когда вызывается

- Соня даёт один абзац / лид / заголовок / блок и просит «поправь»
- Нужна точечная работа, не диагностика всей статьи
- Нужно объяснение каждой правки, чтобы Соня видела логику

## Что подгружает

1. `pr-helper/shared/zerocoder-context.md` (только если фрагмент про бренд)
2. `promts/style-reference.md`
3. `promts/editorial-criteria.md` (антипаттерны и принципы)

---

## Алгоритм работы

### Шаг 1. Маркировать проблемы в исходном фрагменте

Перед правкой — пометить:
- `[нейрослоп]` — где безликий шаблонный текст
- `[не... а]` — клишированные противопоставления
- `[перегруз]` — захламлённые предложения
- `[канцелярит]` — бюрократизмы
- `[пафос]` — абстракции без примеров

### Шаг 2. Исправить точечно

Не переписывать «как могло бы быть в идеале». Минимальные правки, сохраняя авторский замысел и структуру.

### Шаг 3. Объяснить каждую правку

Для каждой замены — короткое пояснение, почему так лучше.

---

## Формат ответа

```
1. Исходный фрагмент с маркерами
   [текст с метками антипаттернов и подчёркнутыми проблемами]

2. Исправленный фрагмент
   [текст после правки]

3. Пояснения к правкам
   - [было] → [стало]. Причина: [...]
   - [было] → [стало]. Причина: [...]
   - ...

4. Что не трогали и почему
   - [фрагмент] — сильное место, оставляем
   - [фрагмент] — серийная отсылка, не трогать
```

---

## Что НЕ делать

- Не возвращать полный текст статьи, если Соня просила только фрагмент. Полный текст — по отдельному запросу.
- Не переписывать смысл — только формулировки.
- Не убирать брендовый блок + анонс, если он в фрагменте.
- Не трогать серийные отсылки («Ранее мы разбирали…»).

## Главная цель

Точечно усилить фрагмент с прозрачным обоснованием каждой правки. Соня должна видеть **почему** каждое изменение лучше — это образовательная функция режима.
