# PR-Helper — мультиагент Сони

Регулярный PR-цикл Зерокодера. 5 специализированных агентов + общая база контекста.

Корневой контекст `клодик/CLAUDE.md` уже подгружен — там роль Сони, общие принципы редактуры, антипаттерны. Здесь — карта системы и инструкции уровня мультиагента.

---

## Карта агентов

| Папка | Агент | Когда вызывать |
|---|---|---|
| `pitch/` | **Pitch-maker** | Нужен питч в издание: на вход тема/черновик + ссылка на СМИ → выход питч с обоснованием темы, тезисами, рекомендациями |
| `b2bchat/` | **B2B-чат** | Нужен пост для TG-канала Зерокодера: либо черновик → пост по референсам, либо «анализ медиаполя → рекомендации → пост» |
| `article-adapter/` | **Адаптер статей** | Готовый материал нужно адаптировать под конкретное издание (стиль, длина, угол, рубрика). НЕ путать с `клодик/adapter/` — тот делает карточки для собственного сайта Зерокодера и в pr-helper не входит |
| `mediabase/` | **CRM-агент** | Работа с таблицей медиа: статусы, касания, ресёрч редакций, контакты, напоминания о 2-м и 3-м касании |
| `newsjacking/` | **Ньюсджекер** | Ежедневный мониторинг медиаполя + TG-каналов → дайджест в Telegram (10:00 МСК) |

---

## Общая база — `shared/`

**Любой контекст о Зерокодере, спикерах, бренде, медиабазе → берётся из `shared/`. Не дублировать в подпапках агентов.**

```
shared/
  zerocoder-context.md     ← компания, спикеры, TOV, клиенты, риск-зоны
```

Если агенту нужен контекст бренда — он читает `pr-helper/shared/zerocoder-context.md` и не создаёт свою копию. При изменениях правится один файл.

Планируется расширить:
- `shared/brand-voice.md` — детальный гайд по стилю и табу
- `shared/speakers/` — досье на каждого спикера
- `shared/references/` — корпоративные ссылки, ключевые сообщения

---

## Как агенты связаны

**Общая БД — единый лист «Медиабаза» через shared-библиотеку `mediabase/sheets.py`:**
- `mediabase/` (CRM-агент) — управляет листом, ведёт ресёрч, обновляет статусы
- `article-adapter/` — читает через `get_adaptation_params(site)`
- `pitch/` — читает через `get_touch_state(site)`, пишет через `log_touch()` (с подтверждения Сони)

**Лист «Медиабаза» — 15 колонок:**
`сайт | slug | название | тип | приоритет | охват | редакция | повестка_ии | контакты | правила_публикации | статус_прогрева | история_касаний | следующее_касание | последний_ресёрч | комментарий`

Приоритет (1-5) считается автоматически по охвату через `sheets.calc_priority()`. Поиск по `сайт` (домен) с нормализацией.

**Поток данных:**
```
newsjacking → дайджест → Соня выбирает тему → pitch-maker делает питч →
  → CRM фиксирует отправку → CRM напоминает о касании →
  → статья принята → адаптер готовит карточку → CRM обновляет статус →
  → b2bchat делает пост с анонсом публикации
```

## Стандарт оформления агента

Чтобы агенты были взаимозаменяемыми и автономными, единый паттерн:

```
pr-helper/<agent>/
  CLAUDE.md              ← кто этот агент, что делает, что читает при старте
  .claude/
    skills/<skill>/SKILL.md
    agents/<sub>.md      ← субагенты, если нужны (как в pitch/)
    agent-memory/        ← локальная память агента
  inputs/                ← входы (черновики, ТЗ)
  outputs/               ← готовые материалы
```

При создании нового агента или скилла — следовать этому паттерну.

## Правила работы

1. **Контекст бренда — только из `shared/`.** Не пересказывать в подпапках.
2. **Файлы данных** — `.csv` для таблиц, `.md` для текстов. `.docx` только если требует издание.
3. **Нейминг** — `[ГГГГ-ММ]_[тема]_[тип].md` (например, `2026-06_vn_metrikokratiya_pitch.md`).
4. **Редактура и стиль** — все принципы в модульных промтах `article-adapter/promts/` (style-reference, editorial-criteria, brand-block-pattern). Антипаттерны (нейрослоп, «не…а», перегруз, канцелярит, пафос) флагировать явно.
5. **Серийные отсылки** в статьях («Ранее мы разбирали…») — намеренный приём, не трогать.
6. **Никаких бартеров.** СМИ с отказом от бартера остаются в базе с пометкой и прорабатываются через другие форматы.
